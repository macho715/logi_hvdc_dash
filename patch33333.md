판정: **예 — UAE 네트워크맵은 `OverviewMap.tsx` 내부 레이어 우선순위와 노드/간선 의미만 재편하면 됩니다. 이번 패치는 `map-only`로 고정하고, `UAE Ops`를 기본값으로 두는 것이 맞습니다.**
근거: 당신이 확정한 와이어프레임 기준에서 UAE Ops는 **`Port/Air → Customs → WH(optional) → MOSB(optional) → Site`** 읽기 순서를 복구하는 것이 핵심이며, **WH 중심 해석·Customs 생략·DAS/AGI direct path**는 금지입니다.
다음행동: **`OverviewMap.tsx`를 `track-aware semantic map`으로 바꾸고, UAE 모드에서는 `customs synthetic nodes + uae semantic edges + subdued trips` 조합으로 패치하십시오.**

# UAE 네트워크맵 패치 방향

## 1) 이번 패치의 목표 한 줄

**새 지도를 만드는 것이 아니라, 현재 OverviewMap을 `UAE Ops semantic network`로 다시 읽히게 만드는 것**입니다.

즉 바꿔야 하는 것은 배경맵이 아니라 아래 4개입니다.

1. 노드 의미
2. 간선 의미
3. 레이어 우선순위
4. tooltip / legend / footer 문구

---

## 2) 범위 고정

### 수정 대상

* `OverviewMap.tsx`
* `MapLegend.tsx`
* `createTripsLayer.ts`
* `POI / node label 관련 layer`
* 필요 시 `customsAnchors.ts`, `uaeOpsGraph.ts` 신규 추가

### 비수정 대상

* `OverviewPageClient` KPI
* `MissionControl`
* `SiteDeliveryMatrix`
* `/api/overview` 구조
* 전체 stage algorithm
* FC ribbon / summary wording

즉 이번은 **화면 전체가 아니라 맵만** 손보는 패치입니다.

---

## 3) UAE Ops의 정답 구조

## 3-1) 읽기 순서

사용자는 UAE 맵을 보고 아래 순서로 이해해야 합니다.

```text
Port / Air
→ Customs
→ Direct or WH
→ MOSB if offshore
→ Site
```

## 3-2) 핵심 원칙

* **WH는 optional staging**
* **Customs는 독립 노드**
* **MOSB는 offshore hub**
* **DAS / AGI는 MOSB 경유 필수**
* **SHU / MIR은 direct 또는 WH 경유 가능**

---

## 4) 레이아웃 패치 규칙

## 4-1) 노드 배치 규칙

UAE 모드에서는 노드가 아래 위치 논리로 고정돼야 합니다.

| 구역     | 배치           | 노드                                       |
| ------ | ------------ | ---------------------------------------- |
| 좌측/좌상단 | Entry        | Khalifa Port, Mina Zayed, AUH Airport    |
| 중앙 상단  | Clearance    | Khalifa Customs, MZ Customs, AUH Customs |
| 중앙     | Staging      | DSV WH                                   |
| 중앙 우측  | Offshore Hub | MOSB                                     |
| 우측     | Final Sites  | SHU, MIR, DAS, AGI                       |

### 금지

* 모든 선이 Abu Dhabi 중앙 한 점으로 수렴하는 구조
* WH가 화면 중심 메인 허브처럼 보이는 구조
* Customs가 label이 아니라 badge 수준으로만 붙는 구조

---

## 4-2) 노드 색상 규칙

색상은 stage가 아니라 **node semantics**로 고정해야 합니다.

| 노드 타입          | 색상     | 의미                      |
| -------------- | ------ | ----------------------- |
| Port / Airport | Blue   | UAE entry               |
| Customs        | Cyan   | clearance               |
| Warehouse      | Yellow | optional inland staging |
| MOSB           | Orange | offshore staging hub    |
| Site           | Green  | SHU/MIR/DAS/AGI         |

### 금지

* stage 상태에 따라 node 색이 바뀌는 방식
* trip 상태 색과 node 타입 색이 섞이는 방식

---

## 4-3) 간선 규칙

UAE 모드에서 의미 있는 간선은 아래만 허용합니다.

| No | Source   | Target  | 설명                           |
| -- | -------- | ------- | ---------------------------- |
| 1  | Port/Air | Customs | 모든 entry의 첫 단계               |
| 2  | Customs  | Site    | SHU/MIR direct 가능            |
| 3  | Customs  | WH      | inland staging               |
| 4  | Customs  | MOSB    | direct offshore staging 필요 시 |
| 5  | WH       | Site    | SHU/MIR staging 후 delivery   |
| 6  | WH       | MOSB    | offshore staging             |
| 7  | MOSB     | DAS/AGI | final offshore leg           |

### 절대 금지

* `Port → WH`
* `Port → MOSB`
* `Port → Site`
* `WH → AGI/DAS` direct
* `Customs` 없는 DAS/AGI route

---

## 4-4) 간선 색상 규칙

| 간선 타입               | 색상         |
| ------------------- | ---------- |
| Port → Customs      | Cyan       |
| Customs → WH        | Blue       |
| Customs → Site      | Light Blue |
| Customs / WH → MOSB | Violet     |
| WH → Site           | Steel Blue |
| MOSB → DAS/AGI      | Green      |

### 구현 팁

애니메이션은 간선마다 다르게 과하게 주지 말고,

* `selected shipment`
* `hovered shipment`
* `Next 72h risky shipment`
  정도만 발광시키는 편이 낫습니다.

# 5) OverviewMap.tsx에서 실제로 바꿔야 할 것

## 5-1) track prop 도입

현재 맵이 `global`과 `uae-ops`를 실제로 구분하지 못하면 안 됩니다.

```ts
type MapTrack = 'global' | 'uae-ops'
```

```tsx
<OverviewMap track={dashMode === 'program' ? 'global' : 'uae-ops'} />
```

### 핵심

* Overview 기본값은 `uae-ops`
* Global은 보조 트랙
* 같은 맵 컴포넌트 안에서 레이어 조합만 바꿔야 함

---

## 5-2) UAE 모드 레이어 스택

UAE Ops에서는 레이어 우선순위를 아래처럼 재편하십시오.

### Z-order 권장

1. dark basemap
2. geofence / terrain tone
3. UAE semantic edge layer
4. customs synthetic node layer
5. semantic POI node layer
6. subdued trips layer
7. selected shipment glow
8. labels / tooltip / footer

### 핵심

**trip line이 주연이 아니라 semantic edge가 주연**이어야 합니다.

---

## 5-3) UAE 모드에서 끄거나 약화할 것

| 항목                        | 처리            |
| ------------------------- | ------------- |
| Origin Arc                | OFF           |
| Heatmap                   | default OFF   |
| Status ring               | opacity down  |
| Location occupancy bubble | priority down |
| dense POI labels          | declutter     |
| `Flow 1/2/3/4` wording    | 삭제            |

# 6) Customs synthetic node 설계

## 6-1) 왜 필요하나

현재 UAE 맵이 창고 중심처럼 읽히는 가장 큰 이유는 **Customs가 중간 backbone으로 안 보이기 때문**입니다.

따라서 customs는 별도 anchor node로 넣어야 합니다.

## 6-2) 최소 customs nodes

* `Khalifa Customs`
* `MZ Customs`
* `AUH Customs`

## 6-3) 좌표 방식

실측 geocode보다 **design anchor** 방식이 맞습니다.

예:

* Khalifa Port 오른쪽 약간 위
* Mina Zayed 오른쪽 약간 위
* AUH Airport 오른쪽

즉 port node 옆 고정 offset.

## 6-4) click action

* customs click → `/chain?focus=customs`
* chain에서 customs focus 미지원이면 fallback `/chain?focus=port`

# 7) Trips layer는 없애지 말고 격하

## 7-1) 왜 유지하나

완전히 빼면 맵이 죽습니다.
하지만 현재처럼 trip이 메인 서사면 또 WH 중심/route classifier처럼 읽힙니다.

## 7-2) 바꿀 방향

Trips layer는 **semantic network 위의 보조 애니메이션**으로 내리십시오.

### 변경 규칙

* line width 축소
* opacity 축소
* default glow 제거
* selected shipment만 강조
* tooltip에서 `Flow Code` 제거

---

## 7-3) tooltip 문구 교체

### 기존 금지

```text
Flow 3
mosbRoute
```

### 교체

```text
Voyage: HVDC-ADOPT-XXXX
Vendor: Prysmian
Route: Customs → WH → MOSB → DAS
Stage: customs_in_progress
Next: WH staging
```

### tooltip 필드 우선순위

1. shipment_id
2. vendor
3. current route text
4. stage
5. next milestone
6. ETA/ATA

# 9) Map footer 방향

이건 매우 중요합니다. KPI를 못 건드려도 map 내부 footer는 의미를 정리해 줍니다.

## UAE footer 권장

```text
Port Pending | Customs Hold | WH Dwell | MOSB Pending | SHU Ready | MIR Ready | DAS Risk | AGI Risk
```

### 최소 버전

데이터가 모자라면 이 버전으로 축소:

```text
Customs Hold | WH Staging | MOSB Pending | Site Ready
```

### 원칙

* site 4개 readiness를 따로 주는 게 좋음
* WH는 summary의 일부이지 중심 지표가 아님
* footer는 숫자보다 상태 읽기 속도가 중요
