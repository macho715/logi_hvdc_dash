# Logistics Dashboard System Architecture

Last updated: 2026-03-15

## Overview

`apps/logistics-dashboard` is a Next.js dashboard that combines:

- logistics map and event visibility
- HVDC-style KPI and worklist monitoring
- cross-page drilldowns driven by a shared URL contract

The current architecture is built around:

- a shared dark premium theme layer
- a dedicated overview backend-for-frontend route
- URL-restored target pages
- Supabase-backed server routes
- shared helper modules for summary aggregation and event mapping

## High-Level Runtime

```text
Browser
  -> Next.js dashboard shell
  -> Overview and deep-link target pages
  -> Shared ops state for worklist and map payloads
  -> URL-restored context chips

Next.js app
  -> Server routes under app/api/*
  -> Overview BFF aggregation
  -> Navigation contract parsing and serialization
  -> Theme token + recipe system

Supabase
  -> v_cases
  -> shipments / shipments_status
  -> locations / location_statuses / events
  -> stock and related operational views
```

## Shell Architecture

Global dashboard shell:

- `app/(dashboard)/layout.tsx`
- `Sidebar`
- `DashboardHeader`
- `KpiProvider`

Behavior:

- `KpiProvider` mounts once at the dashboard layout level
- child pages render inside the same shell
- page styling is composed from shared theme recipes, not page-local theme systems

## Page Architecture

### Overview

Entry:

- `app/(dashboard)/overview/page.tsx`
- `components/overview/OverviewPageClient.tsx`

Data path:

- `useOverviewData()`
- `GET /api/overview`

Layout:

1. toolbar
2. program bar
3. KPI rail
4. ribbon
5. map plus mission control
6. site matrix
7. radar plus ops snapshot

Important behavior:

- map layer toggles live in the toolbar
- map clicks produce `NavigationIntent` drilldowns
- the heatmap layer consumes `map.events` from `/api/overview`

### Deep-link target pages

Pages:

- `/pipeline`
- `/sites`
- `/cargo`
- `/chain`

Pattern:

- `page.tsx` is a server component
- `*PageClient.tsx` owns `useSearchParams()`
- `page.tsx` wraps the client subtree in `Suspense`

Reason:

- avoid static generation failures with URL-driven client state

## Navigation Architecture

Source of truth:

- `lib/navigation/contracts.ts`
- `configs/overview.destinations.json`
- `configs/overview.route-types.json`

Concepts:

- `NavigationIntent`
- per-page query parser and serializer
- destination allowlists and defaults

Public page query contract:

- pipeline: `stage`, `site`, `vendor`, `category`, `route_type`
- sites: `site`, `tab`
- cargo: `tab`, `caseId`, `site`, `vendor`, `voyage_stage`, `route_type`
- chain: `focus`, `site`, `route_type`

Page context chips:

- generated by `getPageContextChips()`
- rendered by `PageContextBanner`

## Data Architecture

### Overview BFF

Route:

- `app/api/overview/route.ts`

Responsibilities:

- aggregate case summary from `v_cases`
- aggregate shipment-stage compliance signals
- load locations, statuses, and events
- build hero, pipeline, route summary, site readiness, alerts, live feed, and warehouse pressure
- return one `OverviewCockpitResponse`

This route is the overview data source of truth.

### Shared summary architecture

Files:

- `lib/cases/summary.ts`
- `app/api/cases/summary/route.ts`
- `app/api/overview/route.ts`

Purpose:

- build fresh cases-summary objects without nested shared mutation
- keep `/api/cases/summary` and `/api/overview` aligned

### Shared event architecture

Files:

- `lib/logistics/events.ts`
- `app/api/events/route.ts`
- `app/api/overview/route.ts`

Purpose:

- own the joined-event Supabase select string
- map Supabase rows into dashboard `Event`
- provide a common mock fallback when live events are unavailable

Why it matters:

- the overview heatmap depends on `map.events`
- `/api/events` and `/api/overview` must not drift to different event query logic

### Other server routes

Main routes:

- `GET /api/cases`
- `GET /api/cases/summary`
- `GET /api/shipments`
- `GET /api/shipments/stages`
- `GET /api/shipments/trips`
- `GET /api/shipments/vendors`
- `POST /api/shipments/new`
- `GET /api/stock`
- `GET /api/locations`
- `GET /api/location-status`
- `GET /api/events`
- `GET /api/chain/summary`

### Shared stores

Overview and page components interact with stores instead of duplicating fetched state.

Examples:

- shared ops store for worklist rows, KPIs, locations, statuses, and events
- cases store for active pipeline stage
- logistics UI store for local map toggles and highlight state

## Realtime Architecture

Global owner:

- `KpiProvider`

Realtime hook:

- `useKpiRealtime()`

Rules:

- one dashboard-level KPI realtime owner
- overview does not create a second global realtime subscription
- overview refresh logic uses page-local polling and focus refetch instead

## Plain-Language Route Model

User-facing route labels come from config, not from hardcoded component strings.

Source:

- `configs/overview.route-types.json`
- `lib/overview/routeTypes.ts`

Public route labels:

- `출발 준비 중`
- `항만 -> 현장 직송`
- `항만 -> 창고 -> 현장`
- `항만 -> MOSB -> 현장`
- `항만 -> 창고 -> MOSB -> 현장`
- `경로 확인 필요`

Internal note:

- `flow_code` still exists as a compatibility field
- public UI contracts are centered on `route_type`

## Deployment Architecture Notes

- Vercel is the primary deployment target
- real-data deployments require valid Supabase environment variables
- placeholder mode is for controlled fallback or parity testing
- pages using `useSearchParams()` must preserve the current `Suspense` split pattern
