판정: **예 — 지금 화면은 `UAE Ops`로 선택돼 있어도 실제로는 `Global origin arc + trip wedge + UAE 내부선`이 동시에 살아 있어서, UAE 네트워크맵으로 읽히지 않습니다. `OverviewMap.tsx`에서 바로 패치해야 합니다.**
근거: 현재 `OverviewMap.tsx`는 `origin arcs`, `heatmap`, `eta wedge`, `location`, `trips`, `status rings`, `poi`를 한 배열로 같이 올리고 있고, trip tooltip도 아직 `Flow ${tooltip.flowCode}`를 노출합니다. 또한 `/api/shipments/trips`와 `/api/chain/summary`를 동시에 받아 same layer stack에 섞어 씁니다.  
다음행동: **`OverviewMap.tsx`를 `track-aware`로 바꾸고, UAE 모드에서는 `origin arcs OFF + customs node ON + semantic edge ON + subdued trips`로 재조합하십시오.**

## 지금 화면에서 확인된 문제

첨부 화면 기준으로 UAE Ops 맵은 아래 문제가 있습니다.

1. **굵은 글로벌 웨지 선이 그대로 살아 있음**
   오른쪽 밖에서 들어오는 대형 선들이 UAE 내부 네트워크를 덮고 있습니다.

2. **모든 흐름이 Abu Dhabi 근처 한 점으로 몰려 보임**
   결과적으로 `WH 중심 허브`처럼 읽힙니다.

3. **Customs가 구조적으로 안 보임**
   좌측 legend에는 있지만 실제 backbone node로 거의 안 읽힙니다.

4. **DAS/AGI offshore logic이 약함**
   MOSB 경유가 “규칙”이 아니라 그냥 여러 선 중 하나처럼 보입니다.

5. **노드보다 선이 너무 강함**
   현재는 `node semantics`보다 `arc mass`가 강해서 control-tower 관계망 해석이 깨집니다.

## 5) UAE semantic edge를 따로 그려야 합니다

현재 `trips`가 주연이라서 선들이 너무 많고, endpoint도 semantics보다 animation 중심입니다. UAE Ops에서는 **semantic edge layer**가 주연이어야 합니다.

### edge 타입

```ts
type UaeEdgeKind =
  | "port_to_customs"
  | "customs_to_wh"
  | "customs_to_site"
  | "customs_to_mosb"
  | "wh_to_site"
  | "wh_to_mosb"
  | "mosb_to_site"
```

### 규칙

* `Port/Air → Customs`
* `Customs → Site`
* `Customs → WH`
* `Customs → MOSB`
* `WH → Site`
* `WH → MOSB`
* `MOSB → DAS/AGI`

### 절대 금지

* `Port → WH`
* `Port → Site`
* `Port → MOSB`
* `WH → DAS/AGI`
* `Customs` 없는 direct offshore
